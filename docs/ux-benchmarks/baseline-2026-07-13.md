# Level CRE UX Baseline - 2026-07-13

## Scope

This baseline covers eight deterministic broker journeys on desktop and mobile plus one deployed Google Maps smoke test. It is a mechanical workflow benchmark, not a score for the entire product or a replacement for Patrick's judgment.

The first runs surfaced an unnamed toast close button, an unnamed Activity status filter, a restrictive viewport setting, repeated low-contrast labels, unclear ownership of shared pursuits, a name-only pursuit setup, nested interactive controls, and a fixture that could intercept the Google Maps script. Those issues were corrected before recording this baseline.

## Team Viability

The scripted evidence supports a small-team pilot. An experienced broker can distinguish owned from shared work, see the junior broker responsible for a pursuit, and understand their access level. An assistant can clean the review queue without turning weak email evidence into a prospect or map pin. A junior broker receives enough structure to anchor a listing farm to a property or market area.

This does not prove broad adoption. The suite uses synthetic data and one browser identity, so real invitations, two-account permissions, onboarding, and live synchronization still need a signed-in team trial.

## Real Map

The deployed map completed an Edmonton Places search, selected an unsaved address, and switched to aerial imagery. The visual gate now waits until map tiles cover the viewport so it cannot accept a half-painted map.

The modern Places request still returns `403 API_KEY_SERVICE_BLOCKED` for `places.googleapis.com`. The legacy compatibility path supplies the visible results, so the user journey works, but the production browser key must explicitly allow Places API (New) before the real-map diagnostic is clean.
